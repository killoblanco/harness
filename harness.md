# AI Harness Definition Manual v3

> **What is this file?** This is a prescriptive instruction set. When an AI agent reads this file, it generates a complete project harness — an externalized state infrastructure within the `.claude/` directory that enables agents to work effectively across multiple sessions despite having no memory between context windows.

## Preamble

### Instructions for the Agent

1. **Read this entire file before taking any action.** Understand the full scope before generating anything.
2. **Run Phase 0 first.** If a `.claude/state/` directory already exists at the project root, abort immediately — do not generate or modify anything.
3. **Language rules:**
   - All generated files, code, and documentation must be written in English.
   - Always respond to the user in the same language they use to communicate with you.
4. **Ask questions freely.** There is no limit on clarifying questions. Ask as many as needed to fully understand the project before generating the harness.
5. **Required information before starting:**
   - Ticket key prefix (e.g., "DS", "AUTH", "PLAT") — used for naming ticket files as `{PREFIX}-001.json`
   - Any existing conventions the user wants preserved
   - Team size and workflow preferences (solo dev, small team, large team)
6. **Dependency:** This harness requires `jq` to be available on the system for JSON manipulation in hook scripts.

### Autonomy Model

This protocol applies to ALL agents generated by this harness, in ALL sessions:

#### Plan Approval + Safeguard

1. **Plan Proposal**: At the start of any task, the agent proposes a complete execution plan: what will be done, which files will be affected, expected outcome.
2. **Approval Gate**: Wait for explicit user approval of the plan. Do NOT proceed until the user confirms.
3. **Autonomous Execution**: After approval, execute the plan autonomously. Save checkpoints continuously (handled automatically by hooks). Do NOT ask for permission at each step.
4. **Mandatory Pause Conditions**: The agent MUST stop and ask the user when:
   - An ambiguity is discovered that wasn't in the approved plan
   - An architectural decision is needed that wasn't contemplated
   - A dependency doesn't work as expected and the approach must change
   - The scope needs to expand or pivot from the approved plan
   - A blocker is encountered that requires human input

#### What Counts as Needing a Pause

- Choosing a library/tool when the plan didn't specify one
- Discovering a file structure different from what was assumed in the plan
- Finding that an API or interface doesn't match expectations
- Needing to modify files outside the approved scope
- Realizing the approach won't work and an alternative is needed

### Task Verification Protocol

Task verification is executed by the **Reviewer agent** (not by the agent that implemented the work). See Appendix D for agent definitions and Appendix F for the full verification protocol.

Summary:
1. Developer marks work as complete
2. Reviewer agent is spawned with clean context
3. Reviewer receives: ticket JSON, acceptance criteria, code diff
4. Reviewer executes: AC review, automated checks (tests/build/lint), E2E validation
5. Reviewer produces: verification report (pass/fail per criterion)
6. If PASS → ticket moves to `done`. If FAIL → specific issues returned to Developer.

### Context Handoff Protocol

When an agent detects its context window is approaching capacity:

1. **Save complete state** to `checkpoint.json`: current position, what's done, what's pending, all decisions made
2. **Update** `progress.json` with current feature statuses
3. **Update** `sessions.json` with partial session data
4. **End session cleanly** with specific, actionable "next steps" in the checkpoint
5. The next session starts with `on-session-start.sh` hook which reads all state and injects it as initial context

The agent should proactively initiate handoff when it estimates context usage is "high" — do not wait until the window is full.

### Definition: AI Harness

An AI harness is externalized state infrastructure that solves the "shift engineer" problem: each new agent session arrives with no memory of previous sessions. The harness creates persistent, machine-readable artifacts that allow each new session to:

1. Rapidly acquire context via automated state injection (hooks restore checkpoint at session start)
2. Continue productive work without repeating discoveries (JSON state tracks exactly where work left off)
3. Coordinate with other agents through shared state files (progress.json, checkpoint.json)
4. Maintain quality standards via independent verification (Reviewer agent with clean context)

### Claude Code Compatibility

This harness operates entirely within `.claude/` and is designed to work with Claude Code's native infrastructure:

| Path | Type | Claude Code Behavior |
|------|------|---------------------|
| `.claude/CLAUDE.md` | Native | Auto-loaded at every session start |
| `.claude/settings.json` | Native | Project configuration + hooks |
| `.claude/agents/*.md` | Native | Auto-discovered as subagent definitions |
| `.claude/rules/*.md` | Native | Path-scoped instructions (loaded on demand) |
| `.claude/hooks/` | Convention | Scripts referenced from settings.json hooks |
| `.claude/state/` | Custom | Ignored by CC — safe for JSON state |
| `.claude/tickets/` | Custom | Ignored by CC — safe for ticket data |
| `.claude/templates/` | Custom | Ignored by CC — safe for schemas |
| `.claude/docs/` | Custom | Ignored by CC — safe for documentation |

**Note:** Claude Code's auto-memory (`~/.claude/projects/<project>/memory/`) is per-user and local. Our `state/` directory is shared team state committed to git — different purpose, no conflict.

## Phase 1: Project Analysis

Before generating any files, analyze the target project to determine what kind of harness to produce.

### Detection Checklist

Execute these checks in order:

1. **Package manager / language detection:**
   - Look for: `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle`, `Gemfile`, `*.csproj`, `Makefile`, `CMakeLists.txt`
   - Determine: primary language, framework, package manager

2. **Complexity assessment:**
   - Count total files and directories (exclude `node_modules`, `.git`, `vendor`, `dist`, `build`)
   - Estimate lines of code
   - Classify:
     - **Small**: <5,000 LOC or <50 files
     - **Medium**: 5,000–50,000 LOC or 50–500 files
     - **Large**: >50,000 LOC or >500 files

3. **Architecture pattern detection:**
   - Check for monorepo indicators: `workspaces` in package.json, `lerna.json`, `/packages/`, `/apps/`, `pnpm-workspace.yaml`
   - Check for microservices: multiple `Dockerfile`s, `docker-compose.yml` with multiple services
   - Check for library patterns: `src/index.ts` or `src/lib/` as sole entry point
   - Classify: `monolith` | `monorepo` | `microservices` | `library`

4. **Test framework detection:**
   - Look for: `jest.config`, `vitest.config`, `pytest.ini`, `conftest.py`, `.rspec`, `*_test.go`, `Cargo.toml [dev-dependencies]`
   - Note: which test runner, where tests live, how to run them

5. **CI/CD detection:**
   - Look for: `.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile`, `.circleci/`, `bitbucket-pipelines.yml`

6. **Existing Claude Code configuration:**
   - Check for: `.claude/CLAUDE.md`, `.claude/settings.json`, `.claude/agents/`, `.claude/rules/`
   - If found: read and incorporate — do NOT overwrite existing configuration

7. **Git history analysis:**
   - Check branch strategy (main/develop, trunk-based, gitflow)
   - Check commit message conventions (conventional commits, etc.)
   - Check recent activity patterns

### Analysis Output

Form a mental model with these attributes (do not write this anywhere — use it to guide generation):

| Attribute | Values |
|-----------|--------|
| `stack` | Primary language + framework (e.g., "TypeScript + Next.js") |
| `complexity_tier` | `small` \| `medium` \| `large` |
| `architecture_pattern` | `monolith` \| `monorepo` \| `microservices` \| `library` |
| `test_framework` | Detected testing setup |
| `has_existing_claude_config` | `true` \| `false` |

```json
{
  "schema": "ticket-v1",
  "key": "{PREFIX}-{NUMBER}",
  "title": "Short descriptive title",
  "type": "story|bug|task|spike",
  "status": "backlog|todo|in_progress|in_review|done|blocked|cancelled",
  "priority": "critical|high|medium|low|trivial",
  "epic": null,
  "sprint": null,
  "assignee": null,
  "reporter": null,
  "storyPoints": null,
  "labels": [],
  "components": [],
  "blockedBy": [],
  "blocks": [],
  "relatedTo": [],
  "dueDate": null,
  "timeBox": null,
  "created": "YYYY-MM-DD",
  "updated": "YYYY-MM-DD",
  "resolved": null,
  "resolution": null,
  "description": "Full description of the work",
  "acceptanceCriteria": [
    {
      "id": 1,
      "description": "What must be true",
      "verified": false,
      "verifiedBy": null,
      "verifiedAt": null
    }
  ],
  "technicalNotes": "Implementation details, gotchas, architecture notes",
  "comments": [
    {
      "date": "ISO-8601",
      "author": "agent|user|reviewer",
      "content": "Comment text"
    }
  ]
}
```

| Field | Type | Required | Description |

| `status` | enum | yes | `backlog` \| `todo` \| `in_progress` \| `in_review` \| `done` \| `blocked` \| `cancelled` |
| `priority` | enum | yes | `critical` \| `high` \| `medium` \| `low` \| `trivial` |
| `epic` | string | no | Epic identifier |
| `sprint` | string | no | Sprint name |
| `assignee` | string | no | Who is working on this |
| `reporter` | string | no | Who created this |
| `storyPoints` | integer | no | Effort estimate |
| `labels` | array | no | Tags for categorization |
| `components` | array | no | System components affected |
| `blockedBy` | array | no | Ticket keys that block this |
| `blocks` | array | no | Ticket keys this blocks |
| `relatedTo` | array | no | Related ticket keys |
| `dueDate` | string | no | Deadline (YYYY-MM-DD) |
| `timeBox` | string | no | For spikes: "{N}h" |
| `created` | string | yes | Creation date (YYYY-MM-DD) |
| `updated` | string | yes | Last update date (YYYY-MM-DD) |
| `resolved` | string | no | Resolution date |
| `resolution` | enum | no | `fixed` \| `wont_fix` \| `duplicate` \| `cannot_reproduce` |
| `description` | string | yes | Full description |
| `acceptanceCriteria` | array | no | Verifiable criteria |
| `technicalNotes` | string | no | Implementation details |
| `comments` | array | no | Timestamped comments |

### Type-Specific Templates

Templates in `.claude/templates/` provide pre-filled structures for each type:

**story.json** — Adds `acceptanceCriteria` with empty criteria list
**bug.json** — Adds `stepsToReproduce`, `expectedBehavior`, `actualBehavior`, `environment` fields
**task.json** — Adds `definitionOfDone` array instead of `acceptanceCriteria`
**spike.json** — Adds `researchQuestion`, `scope`, `findings`, `recommendation` fields; requires `timeBox`

## Appendix B: State File Schemas

### checkpoint.json

The most critical state file. Always reflects the agent's current position. Updated automatically by hooks after every tool use.

```json
{
  "schema": "checkpoint-v1",
  "lastUpdated": "2024-01-15T14:30:00Z",
  "sessionId": "session-uuid",
  "currentAgent": "developer",
  "currentTicket": "{PREFIX}-001",
  "phase": "implement",
  "plan": {
    "approved": true,
    "summary": "Implement JWT auth with refresh tokens",
    "approvedAt": "2024-01-15T14:00:00Z"
  },
  "subtasks": [
    { "id": 1, "description": "Create auth middleware", "status": "done" },
    { "id": 2, "description": "Implement token refresh endpoint", "status": "in_progress" },
    { "id": 3, "description": "Add auth tests", "status": "pending" }
  ],
  "nextSteps": [
    "Complete refresh endpoint implementation",
    "Write integration tests for token flow"
  ],
  "blockers": [],
  "filesModified": ["src/middleware/auth.ts", "src/routes/auth.ts"],
  "uncommittedChanges": true,
  "contextUsage": "medium"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `schema` | string | Always "checkpoint-v1" |
| `lastUpdated` | ISO-8601 | When this checkpoint was last saved |
| `sessionId` | string | Unique identifier for the current session |
| `currentAgent` | enum | `orchestrator` \| `developer` \| `reviewer` |
| `currentTicket` | string | Ticket key being worked on |
| `phase` | enum | `planning` \| `research` \| `implement` \| `test` \| `review` |
| `plan` | object | The approved execution plan |
| `subtasks` | array | Breakdown of current work with statuses |
| `nextSteps` | array | What to do next (critical for handoff) |
| `blockers` | array | Anything preventing progress |
| `filesModified` | array | Files changed since last commit |
| `uncommittedChanges` | boolean | Whether there are unsaved git changes |
| `contextUsage` | enum | `low` \| `medium` \| `high` \| `critical` |

### progress.json

Tracks all features/tickets and their lifecycle status.

```json
{
  "schema": "progress-v1",
  "currentSprint": "sprint-3",
  "features": [
    {
      "key": "{PREFIX}-001",
      "title": "User authentication",
      "status": "in_progress",
      "assignee": "developer",
      "sessions": [1, 2, 3],
      "blockedBy": [],
      "notes": "JWT implementation chosen per ADR-001",
      "startedAt": "2024-01-14",
      "updatedAt": "2024-01-15"
    }
  ]
}
```

**Status flow:** `not_started` → `in_progress` → `in_review` → `complete` (or `blocked` at any point)

Never skip a status. Never go from `not_started` directly to `complete`.

### sessions.json

Log of all agent sessions. Updated automatically by hooks at session end.

```json
{
  "schema": "sessions-v1",
  "sessions": [
    {
      "id": 1,
      "date": "2024-01-15",
      "agent": "developer",
      "objective": "Implement auth middleware",
      "outcome": "Middleware complete, refresh endpoint 60% done",
      "nextSteps": ["Complete refresh endpoint", "Write tests"],
      "ticketsWorked": ["{PREFIX}-001"],
      "decisionsRecorded": [1],
      "checkpointCount": 12,
      "duration": "~45 min"
    }
  ]
}
```

### decisions.json

Architectural decisions in ADR-lite format.

```json
{
  "schema": "decisions-v1",
  "decisions": [
    {
      "id": 1,
      "title": "Use JWT for authentication",
      "date": "2024-01-14",
      "status": "accepted",
      "context": "Need stateless auth for horizontal scaling",
      "decision": "JWT with short-lived access tokens (15m) and long-lived refresh tokens (7d)",
      "consequences": "Stateless scaling possible, but token revocation requires blocklist",
      "alternatives": ["Session-based auth", "OAuth2 only"],
      "relatedTickets": ["{PREFIX}-001"]
    }

## Appendix C: Hook Scripts Reference

### on-session-start.sh

Restores state and injects context at the beginning of every session.

```bash
#!/bin/bash
set -euo pipefail

STATE_DIR="${CLAUDE_PROJECT_DIR}/.claude/state"
CHECKPOINT="${STATE_DIR}/checkpoint.json"
PROGRESS="${STATE_DIR}/progress.json"
SESSIONS="${STATE_DIR}/sessions.json"

# Build context summary for Claude
OUTPUT="{}"

# Add checkpoint context (if exists and non-null)
if [ -f "$CHECKPOINT" ] && [ "$(jq -r '.lastUpdated' "$CHECKPOINT")" != "null" ]; then
  CURRENT_TICKET=$(jq -r '.currentTicket // "none"' "$CHECKPOINT")
  PHASE=$(jq -r '.phase // "none"' "$CHECKPOINT")
  NEXT_STEPS=$(jq -r '.nextSteps | join(", ")' "$CHECKPOINT")
  BLOCKERS=$(jq -r '.blockers | join(", ")' "$CHECKPOINT")
  CONTEXT_USAGE=$(jq -r '.contextUsage // "unknown"' "$CHECKPOINT")
  UNCOMMITTED=$(jq -r '.uncommittedChanges' "$CHECKPOINT")

  OUTPUT=$(echo "$OUTPUT" | jq \
    --arg ticket "$CURRENT_TICKET" \
    --arg phase "$PHASE" \
    --arg next "$NEXT_STEPS" \
    --arg blockers "$BLOCKERS" \
    --arg context "$CONTEXT_USAGE" \
    --arg uncommitted "$UNCOMMITTED" \
    '. + {
      checkpoint: {
        currentTicket: $ticket,
        phase: $phase,
        nextSteps: $next,
        blockers: $blockers,
        previousContextUsage: $context,
        uncommittedChanges: ($uncommitted == "true")
      }
    }')
fi

# Add last 3 sessions summary
if [ -f "$SESSIONS" ]; then
  RECENT=$(jq '.sessions | .[-3:] | map({id, date, agent, outcome, nextSteps})' "$SESSIONS")
  OUTPUT=$(echo "$OUTPUT" | jq --argjson recent "$RECENT" '. + {recentSessions: $recent}')
fi

# Add in-progress features
if [ -f "$PROGRESS" ]; then
  IN_PROGRESS=$(jq '.features | map(select(.status == "in_progress" or .status == "blocked"))' "$PROGRESS")
  OUTPUT=$(echo "$OUTPUT" | jq --argjson features "$IN_PROGRESS" '. + {activeFeatures: $features}')
fi

# Output context for Claude
echo "$OUTPUT" | jq '{
  hookSpecificOutput: {
    hookEventName: "SessionStart",
    additionalContext: (. | tostring)
  }
}'
```

### on-checkpoint.sh

Saves state after every tool use (Edit, Write, Bash). Designed to be lightweight.

```bash
#!/bin/bash
set -euo pipefail

STATE_DIR="${CLAUDE_PROJECT_DIR}/.claude/state"
CHECKPOINT="${STATE_DIR}/checkpoint.json"
ACTION="${1:-post}"

if [ "$ACTION" = "post" ]; then
  # Update timestamp
  TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

  # Get modified files (if git is available)
  FILES_MODIFIED="[]"
  if command -v git &> /dev/null && git rev-parse --git-dir &> /dev/null 2>&1; then
    FILES_MODIFIED=$(git status --porcelain 2>/dev/null | awk '{print $2}' | jq -R -s 'split("\n") | map(select(length > 0))' 2>/dev/null || echo "[]")
  fi

  # Check for uncommitted changes
  UNCOMMITTED=false
  if command -v git &> /dev/null && git rev-parse --git-dir &> /dev/null 2>&1; then
    if [ -n "$(git status --porcelain 2>/dev/null)" ]; then
      UNCOMMITTED=true
    fi
  fi

  # Update checkpoint
  if [ -f "$CHECKPOINT" ]; then
    jq \
      --arg ts "$TIMESTAMP" \
      --argjson files "$FILES_MODIFIED" \
      --argjson uncommitted "$UNCOMMITTED" \
      '.lastUpdated = $ts | .filesModified = $files | .uncommittedChanges = $uncommitted' \
      "$CHECKPOINT" > "${CHECKPOINT}.tmp" && mv "${CHECKPOINT}.tmp" "$CHECKPOINT"
  fi
fi
```

### on-session-end.sh

Persists the complete session record when the session terminates.

```bash
#!/bin/bash
set -euo pipefail

STATE_DIR="${CLAUDE_PROJECT_DIR}/.claude/state"
CHECKPOINT="${STATE_DIR}/checkpoint.json"
SESSIONS="${STATE_DIR}/sessions.json"

TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
TODAY=$(date -u +"%Y-%m-%d")

# Read current checkpoint for session summary
if [ -f "$CHECKPOINT" ] && [ "$(jq -r '.sessionId' "$CHECKPOINT")" != "null" ]; then
  SESSION_ID=$(jq -r '.sessionId' "$CHECKPOINT")
  AGENT=$(jq -r '.currentAgent // "unknown"' "$CHECKPOINT")
  TICKET=$(jq -r '.currentTicket // "none"' "$CHECKPOINT")
  NEXT_STEPS=$(jq '.nextSteps' "$CHECKPOINT")

  # Calculate next session id
  NEXT_ID=$(jq '.sessions | length + 1' "$SESSIONS")

  # Append session entry
  jq \
    --argjson id "$NEXT_ID" \
    --arg date "$TODAY" \
    --arg agent "$AGENT" \
    --arg ticket "$TICKET" \
    --argjson nextSteps "$NEXT_STEPS" \
    '.sessions += [{
      id: $id,
      date: $date,
      agent: $agent,
      objective: "See checkpoint for details",
      outcome: "Session ended",
      nextSteps: $nextSteps,
      ticketsWorked: [$ticket],
      decisionsRecorded: [],
      checkpointCount: 0,
      duration: "unknown"
    }]' \
    "$SESSIONS" > "${SESSIONS}.tmp" && mv "${SESSIONS}.tmp" "$SESSIONS"
fi
```

## Appendix D: Agent Definitions

### Orchestrator Agent

```markdown
# Agent: Orchestrator

## Role
Lead coordination agent responsible for decomposing complex tasks, creating execution plans, assigning work to Developer and Reviewer agents, and resolving conflicts. Does NOT implement code directly.

## When to Use
- Any task that requires planning before implementation
- When work needs to be decomposed into subtasks
- When multiple tickets need prioritization
- When conflicts between agent outputs need resolution

## Capabilities
- Decompose tasks into concrete subtasks with clear boundaries
- Create execution plans with scope, files, and expected outcomes
- Assign work to Developer with explicit scope and context
- Trigger Reviewer for verification after implementation
- Update progress.json and decisions.json
- Manage ticket lifecycle (create, update status, prioritize)

## Constraints
- Does NOT write implementation code
- Does NOT make architectural decisions unilaterally (follows Autonomy Model — proposes, waits for approval)
- Must respect file/module boundaries when assigning work
- Must ensure Developer receives clear, unambiguous instructions

## Quality Checklist
Before considering coordination work complete:
- [ ] Plan was proposed and approved by user
- [ ] Subtasks have clear scope and expected outcomes
- [ ] Developer received all necessary context
- [ ] Reviewer was triggered after implementation
- [ ] progress.json reflects current state
- [ ] Any decisions were recorded in decisions.json
```

### Developer Agent

```markdown
# Agent: Developer

## Role
Implementation agent that writes code following an approved plan. Executes autonomously within the approved scope, pausing only for unexpected situations that require human input.

## When to Use
- Implementing features from approved plans
- Fixing bugs with clear reproduction steps
- Writing tests for new or changed behavior
- Refactoring within approved scope

## Capabilities
- Write, modify, and delete code files
- Run tests and build commands
- Read documentation and existing code for context
- Update technicalNotes on tickets with implementation details

## Constraints
- MUST NOT auto-commit (commits are user decisions)
- MUST work within the scope approved in the plan
- MUST pause for: ambiguities, unplanned architecture decisions, blockers, scope changes
- MUST NOT mark tickets as done (Reviewer does verification)
- MUST NOT modify state files manually (hooks handle checkpoint.json)
- SHOULD update progress.json when starting/blocking on a ticket

## Execution Flow
1. Receive plan (from Orchestrator or directly from user)
2. Propose plan if not already approved → wait for approval
3. Execute plan autonomously, one subtask at a time
4. Run tests after each logical change
5. When all subtasks complete: signal ready for review

## Quality Checklist
Before signaling work as ready for review:
- [ ] All subtasks in the plan are implemented
- [ ] Tests pass (existing + new)
- [ ] Build succeeds
- [ ] Lint is clean
- [ ] No hardcoded secrets or credentials
- [ ] technicalNotes updated on ticket with implementation details
```

### Reviewer Agent

```markdown
# Agent: Reviewer

## Role
Independent verification agent spawned with clean context. Validates that implementation meets acceptance criteria without the bias of having written the code. Executes the full verification protocol.

## When to Use
- After Developer signals work is ready for review
- Before any ticket is moved to "done" status
- When verifying a bug fix actually resolves the issue

## Capabilities
- Read and analyze code changes (diffs)
- Run test suites, build, and lint
- Use Chrome DevTools MCP for E2E validation (when applicable)
- Produce structured verification reports
- Reject work with specific, actionable feedback

## Constraints
- Receives ONLY: ticket JSON, acceptance criteria, code diff, and relevant file context
- Does NOT receive the Developer's conversation history (clean context = unbiased review)
- Does NOT modify implementation code (returns feedback to Developer instead)
- Does NOT approve its own work (never reviews code it wrote)

## Verification Sequence
1. Read each Acceptance Criterion from the ticket
2. For EACH criterion:
   - Verify in code (logic correct, test covers it)
   - Verify E2E if applicable (navigate, interact, observe)
3. Run automated checks:
   - Tests pass
   - Build succeeds
   - Lint clean
4. Check for regressions (did existing tests break?)
5. Flag non-automatable criteria:
   ```
   REQUIRES HUMAN VERIFICATION:
   - AC #3: "Visual design matches mockup" — cannot verify programmatically
   ```
6. Produce verification report

## Output Format
```json
{
  "ticket": "{PREFIX}-001",
  "verdict": "pass|fail",
  "criteria": [
    { "id": 1, "description": "...", "status": "pass|fail|needs_human", "evidence": "..." }
  ],
  "automatedChecks": {
    "tests": "pass|fail",
    "build": "pass|fail",
    "lint": "pass|fail"
  },
  "issues": [
    { "severity": "blocker|major|minor", "description": "...", "file": "...", "suggestion": "..." }
  ],
  "humanVerificationNeeded": ["AC #3: Visual design matches mockup"]
}
```

## Quality Checklist
- [ ] Every AC has been individually assessed
- [ ] Automated checks (tests, build, lint) have been run
- [ ] E2E validation completed (or marked as not applicable)
- [ ] Verification report produced with evidence
- [ ] Issues include specific file references and suggestions
```

## Appendix F: Protocols Reference

### Protocol 1: Autonomy Model

```
┌─────────────────────────────────────────────┐
│ 1. Agent proposes execution plan            │
│    - What will be done                      │
│    - Which files affected                   │
│    - Expected outcome                       │
│    - Estimated scope                        │
├─────────────────────────────────────────────┤
│ 2. User approves (or requests changes)      │
├─────────────────────────────────────────────┤
│ 3. Agent executes AUTONOMOUSLY              │
│    - Hooks save checkpoints automatically   │
│    - Agent updates progress on status       │
│      changes                                │
│    - Agent records decisions when made      │
├─────────────────────────────────────────────┤
│ 4. PAUSE CONDITIONS (must stop and ask):    │
│    - Ambiguity not in plan                  │
│    - Unplanned architecture decision        │
│    - Dependency doesn't work as expected    │
│    - Scope needs to change                  │
│    - Blocker requiring human input          │
└─────────────────────────────────────────────┘
```

### Protocol 2: Context Handoff

When context usage reaches "high":

```
┌─────────────────────────────────────────────┐
│ 1. DETECT: Agent estimates context is full  │
├─────────────────────────────────────────────┤
│ 2. SAVE: Update checkpoint.json with:       │
│    - Exact current position in subtasks     │
│    - All decisions made this session        │
│    - Files modified                         │
│    - Clear "next steps" for continuation    │
├─────────────────────────────────────────────┤
│ 3. UPDATE: progress.json + sessions.json    │
├─────────────────────────────────────────────┤
│ 4. END: Terminate session cleanly           │
│    - "Next steps" must be specific enough   │
│      for a fresh agent to continue          │
│      immediately without re-reading code    │
├─────────────────────────────────────────────┤
│ 5. RESTORE: Next session starts via hook    │
│    - on-session-start.sh reads all state    │
│    - Injects context summary automatically  │
│    - New agent picks up where previous left │
└─────────────────────────────────────────────┘
```

### Protocol 3: Verification (via Reviewer Agent)

```
┌─────────────────────────────────────────────┐
│ 1. Developer signals "ready for review"     │
├─────────────────────────────────────────────┤
│ 2. Reviewer spawned with CLEAN context      │
│    Receives only:                           │
│    - Ticket JSON (acceptance criteria)      │
│    - Code diff (what changed)              │
│    - Relevant file context                  │
├─────────────────────────────────────────────┤
│ 3. Reviewer executes verification:          │
│    - Check each AC individually             │
│    - Run tests, build, lint                 │
│    - E2E validation (if applicable)         │
│    - Flag non-automatable criteria          │
├─────────────────────────────────────────────┤
│ 4. Reviewer produces report:                │
│    - PASS → ticket moves to "done"          │
│    - FAIL → specific issues + suggestions   │
│             returned to Developer            │
├─────────────────────────────────────────────┤
│ 5. If FAIL:                                 │
│    - Developer receives issue list          │
│    - Developer fixes and re-signals review  │
│    - Loop until PASS or user intervenes     │
└─────────────────────────────────────────────┘
```

### Protocol 4: Error Recovery

| Situation | Action |
|-----------|--------|
| Blocked by external dependency | Document in checkpoint.json blockers, update ticket status to `blocked`, move to next task |
| Context window filling | Execute Context Handoff Protocol immediately |
| Tests break unexpectedly | Stop current work, investigate on clean state, decide whether to fix or escalate |
| Conflicting with another agent | Stop, document conflict in checkpoint, escalate to user |
| Stuck in a loop (3 failed attempts) | Document what was tried, mark as blocked, move on |
| Hook script fails | Log error, continue without checkpoint (non-blocking) |

### Protocol 5: Progress Tracking Rules

| File | When to Update | Who Updates |
|------|---------------|-------------|
| `checkpoint.json` | After every tool use | Hooks (automatic) |
| `progress.json` | When any ticket status changes | Agent (manual) |
| `decisions.json` | When any architectural choice is made | Agent (manual) |
| `sessions.json` | At session end | Hook (automatic) |

Rules:
- `sessions.json` is append-only (never edit past sessions)
- `progress.json` features are append-only (but status can be updated)
- `decisions.json` is append-only (status can change to deprecated/superseded)
- `checkpoint.json` is always overwritten (represents current state only)

## Summary

This manual enables any AI agent to:
1. **Analyze** a software project's stack, complexity, and architecture
2. **Generate** a complete harness within `.claude/` leveraging native Claude Code infrastructure
3. **Configure** hooks for automatic, invisible checkpointing
4. **Initialize** JSON state files for reliable, corruption-resistant persistence
5. **Define** three specialized agents (Orchestrator, Developer, Reviewer) for the plan-build-verify cycle
6. **Verify** the harness is correct and functional

The harness then serves as the persistent infrastructure that makes long-running, multi-session AI development possible — solving the fundamental problem of agents with no memory between context windows, while leveraging Claude Code's native capabilities for maximum integration.
